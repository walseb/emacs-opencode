#+TITLE: Emacs OpenCode Architecture

* Overview
This package is an Emacs-native client for OpenCode. It starts or attaches to an
OpenCode server for a project, opens sessions in dedicated Emacs buffers, sends
prompts and commands over HTTP, and reflects server events back into the buffer
as a streaming conversation log.

OpenCode remains the authority for sessions, messages, tools, providers, and
agent behavior. The Emacs side is a thin orchestration and presentation layer:
it owns process lifecycle, request dispatch, event routing, rendering, and the
small amount of local state needed for an interactive editor UI.

* Core Principles
- Emacs-first UX: native buffers, commands, and keybindings.
- Server-first data: OpenCode remains authoritative for sessions and messages.
- Thin orchestration: the client coordinates flows without reimplementing core
  server behavior.
- Async everywhere: server communication is callback-driven and must not block
  editing.
- Incremental rendering: update the smallest affected region and preserve user
  point/scroll state.
- Small, intentional state: cache only what improves responsiveness or display.

* Runtime Model
An OpenCode connection represents one server process for one normalized project
directory. Connections are registered and reused so commands in the same project
share a server, caches, and event stream. If the configured port is unavailable,
the client chooses another free port for that server.

Most interactive commands follow the same shape:

1. Resolve a project directory.
2. Start or reuse the directory's connection.
3. Make an HTTP request to create, select, or update a session.
4. Open or reuse a session buffer.
5. Let the SSE stream drive subsequent UI updates.

There is also a separate one-shot path for running an OpenCode prompt as a CLI
process. That path does not create a session buffer and is intentionally outside
the server-backed session lifecycle.

* Layers
- Entry commands choose directories, manage connection reuse, create or select
  sessions, and bridge interactive commands into the lower layers.
- Connection lifecycle owns server process startup, port selection, readiness,
  health checks, environment variables, shutdown, and connection-scoped caches.
- HTTP client wrappers centralize request construction, JSON encoding, parsing,
  auth, timeouts, and directory routing headers.
- Streaming transport owns the SSE subprocess, event filtering, parsing, and
  dispatch into registered event handlers.
- Session domain models normalize server payloads into small Emacs Lisp structs
  for sessions, messages, message parts, and status.
- Session buffers own input handling, command classification, completion,
  selected agent/model/variant, and history loading.
- Rendering turns buffer-local session state into read-only text with markers,
  text properties, overlays, fontification, collapsible output, and interactive
  task/subagent regions.
- Event handlers translate server events into buffer-local state changes,
  minibuffer prompts, file reloads, and targeted re-renders.

* State Ownership
| Owner           | State                                                        |
|-----------------+--------------------------------------------------------------|
| OpenCode server | Authoritative sessions, messages, tools, providers, agents   |
| Connection      | Server process, base URL, directory, SSE process, caches     |
| Session buffer  | Current session, messages, status, input, selected model data |
| Global registry | Directory->connection and session->buffer lookup tables      |
| Rendered text   | Markers, text properties, overlays, and invisibility state    |

Connection caches are shared across buffers attached to the same server. Active
agent, model, and variant selections are buffer-local because each session can
use different settings. Task/subagent tracking records which parent task tool
created each child session, so events from a child session can update the task
block displayed in the parent session buffer.

* Session Rendering Invariants
- The editable input region is always at the end of the buffer.
- Rendered messages live before the input region and are read-only.
- Each rendered message has stable start/end markers so streaming updates can
  replace only that message's region.
- Rendering must not consume typed input when new output arrives.
- Window point and scroll position should be preserved for users reading older
  output; windows already following the bottom should continue following it.
- Collapsed tool output is keyed by message-part identity so user expansion can
  survive re-renders.
- Header-line and retry banner state are derived from session status and message
  metadata, not from independent persistent UI state.

* Streaming And Event Routing
SSE is the primary mechanism for keeping buffers current after a request is
queued. The preferred streaming backend parses and filters events outside Emacs,
while the fallback backend parses the stream in Emacs. Both paths share the same
event registry and avoid parsing or retaining events that have no registered
handler.

Handlers receive the originating connection as metadata. That matters when more
than one OpenCode server is running: permission replies, question replies, and
subagent updates must be routed back to the server that produced the event, not
to an arbitrary live buffer.

Event handlers should do as little as possible: normalize payloads, find the
right buffer or connection, update local state, and trigger the smallest useful
render. Minibuffer prompts are deferred out of process filters so stream
handling does not block on user input.

* Extension Guide
- To add an HTTP endpoint, add a client wrapper first, then call it from the
  entry, session, or model layer that owns the user-facing behavior.
- To add an SSE event, register a handler, route by session ID or originating
  connection, update buffer-local state, and render only affected UI.
- To add a rendered tool type, extend tool rendering while preserving read-only
  properties, message markers, and any interactive text properties.
- To add input completion, add a completion provider rather than hardcoding the
  trigger into the mode.
- To add provider, model, agent, or auth behavior, keep catalogs and expensive
  lookups connection-scoped; keep the user's active selection buffer-local.
- To add new global state, first check whether it really belongs on the
  connection or in the session buffer. Prefer the narrowest owner that works.

* Failure And Recovery
Server startup fails fast if readiness or health checks do not complete in time.
Session buffers can reconnect by starting a new server for their session
directory and updating their buffer-local connection. Process output stays in
dedicated hidden buffers so UI buffers remain focused on user-facing session
state. User-visible failures should use clear Emacs messages or errors, and
processes/buffers should be cleaned up when a connection shuts down.
